# Standard FE-001 — TanStack Query Cache Invalidation Standard

This standard defines the canonical approach for cache invalidation in RealtorNet using TanStack Query.

## Principle

**Invalidate what you mutate, nothing more, nothing less.**

Every mutation must invalidate the exact queries that are affected by the data change.

## Invalidation Rules

### 1. Invalidation Scope

#### Precise Invalidation (Preferred)
```tsx
// ✅ Correct: Invalidate only the affected query
onSuccess: async () => {
  await queryClient.invalidateQueries({ queryKey: ["myAgencyJoinRequests"] });
}
```

#### Broad Invalidation (When Necessary)
```tsx
// ✅ Correct: Invalidate multiple related queries when mutation affects multiple surfaces
onSuccess: async () => {
  await Promise.all([
    queryClient.invalidateQueries({ queryKey: ["myAgencyJoinRequests"] }),
    queryClient.invalidateQueries({ queryKey: ["myAgencyMemberships"] }),
  ]);
}
```

#### Anti-Pattern: Over-Invalidation
```tsx
// ❌ Incorrect: Invalidating entire cache or unrelated queries
onSuccess: async () => {
  await queryClient.invalidateQueries(); // Invalidates everything
}
```

### 2. Query Key Convention

Query keys must follow a consistent hierarchy:

```tsx
// Format: [resource, identifier?, scope?]
["agencies"]              // All agencies
["agencies", agencyId]    // Specific agency
["agencies", agencyId, "memberships"] // Agency memberships
["myAgencyJoinRequests"]  // Current user's join requests
["agencyJoinRequests", agencyId, "all"] // All join requests for an agency
```

### 3. Invalidation Timing

#### Async Invalidation (Preferred)
```tsx
// ✅ Correct: Wait for invalidation to complete
onSuccess: async () => {
  await queryClient.invalidateQueries({ queryKey: ["myAgencyJoinRequests"] });
}
```

#### Fire-and-Forget (Acceptable for Non-Critical)
```tsx
// ✅ Acceptable: Non-critical invalidation
onSuccess: () => {
  queryClient.invalidateQueries({ queryKey: ["myAgencyJoinRequests"] });
}
```

## Common Patterns

### Pattern 1: Self-Referential Mutation
When a mutation affects its own query:

```tsx
export function useCreateAgencyJoinRequest(agencyId: string | number) {
  const queryClient = useQueryClient();

  return useMutation({
    mutationFn: (payload: AgencyJoinRequestCreate) =>
      apiClient<AgencyJoinRequestResponse>(
        `/api/v1/agencies/${agencyId}/join-request`,
        {
          method: "POST",
          body: JSON.stringify(payload),
        },
      ),
    onSuccess: async () => {
      await queryClient.invalidateQueries({ queryKey: ["myAgencyJoinRequests"] });
    },
  });
}
```

### Pattern 2: Cross-Resource Mutation
When a mutation affects multiple resources:

```tsx
export function useApproveAgencyJoinRequest(agencyId?: string | number | null) {
  const queryClient = useQueryClient();

  return useMutation({
    mutationFn: (joinRequestId: number) =>
      apiClient<AgencyJoinRequestResponse>(
        `/api/v1/agencies/${agencyId}/join-requests/${joinRequestId}/approve`,
        {
          method: "POST",
        },
      ),
    onSuccess: async () => {
      await Promise.all([
        queryClient.invalidateQueries({ queryKey: ["agencyJoinRequests", agencyId, "all"] }),
        queryClient.invalidateQueries({ queryKey: ["myAgencyJoinRequests"] }),
        queryClient.invalidateQueries({ queryKey: ["myAgencyMemberships"] }),
      ]);
    },
  });
}
```

### Pattern 3: List + Detail Mutation
When a mutation affects both list and detail views:

```tsx
export function useUpdateProperty(propertyId: number) {
  const queryClient = useQueryClient();

  return useMutation({
    mutationFn: (payload: PropertyUpdate) =>
      apiClient<Property>(
        `/api/v1/properties/${propertyId}`,
        {
          method: "PUT",
          body: JSON.stringify(payload),
        },
      ),
    onSuccess: async () => {
      await Promise.all([
        queryClient.invalidateQueries({ queryKey: ["properties"] }),
        queryClient.invalidateQueries({ queryKey: ["property", propertyId] }),
        queryClient.invalidateQueries({ queryKey: ["myProperties"] }),
      ]);
    },
  });
}
```

### Pattern 4: Status Change Mutation
When a mutation changes status across multiple views:

```tsx
export function useVerifyProperty(propertyId: number) {
  const queryClient = useQueryClient();

  return useMutation({
    mutationFn: () =>
      apiClient<Property>(
        `/api/v1/properties/${propertyId}/verify`,
        {
          method: "POST",
        },
      ),
    onSuccess: async () => {
      await Promise.all([
        queryClient.invalidateQueries({ queryKey: ["properties"] }),
        queryClient.invalidateQueries({ queryKey: ["property", propertyId] }),
        queryClient.invalidateQueries({ queryKey: ["myProperties"] }),
        queryClient.invalidateQueries({ queryKey: ["featuredProperties"] }),
        queryClient.invalidateQueries({ queryKey: ["agencyProperties"] }),
      ]);
    },
  });
}
```

## Query Key Registry

### Agency Resources
```tsx
["agencies"]                           // Agency directory
["agencies", agencyId]                // Agency detail
["agencies", agencyId, "memberships"] // Agency memberships
["agencies", agencyId, "agents"]       // Agency agents
["agencyJoinRequests", agencyId, "all"] // Agency join requests
["myAgencyJoinRequests"]               // Current user's join requests
["myAgencyInvitations"]                // Current user's invitations
["myAgencyMemberships"]                // Current user's memberships
["agencyApplications"]                 // Agency applications
```

### Property Resources
```tsx
["properties"]                        // Property directory
["property", propertyId]               // Property detail
["myProperties"]                       // Current user's properties
["agencyProperties", agencyId]         // Agency's properties
["featuredProperties"]                // Featured properties
["propertyTypes"]                      // Property types
```

### Inquiry Resources
```tsx
["inquiries"]                         // Inquiry directory
["inquiry", inquiryId]                // Inquiry detail
["myInquiries"]                       // Current user's inquiries
```

```tsx
["users"]                             // User directory

```

## Invalidation Checklist

Before implementing cache invalidation:

- [ ] Identify all queries that display the mutated data
- [ ] Identify all queries that depend on the mutated data
- [ ] Use precise query keys from the registry
- [ ] Use `Promise.all` for multiple invalidations
- [ ] Use `await` for critical invalidations
- [ ] Test that affected surfaces refresh correctly

## Anti-Patterns

### Invalidating Everything
```tsx
// ❌ Incorrect: Invalidates entire cache
onSuccess: async () => {
  await queryClient.invalidateQueries();
}
```

### Missing Invalidation
```tsx
// ❌ Incorrect: No invalidation after mutation
onSuccess: async () => {
  // No invalidation - stale data persists
}
```

### Incorrect Query Key
```tsx
// ❌ Incorrect: Wrong query key
onSuccess: async () => {
  await queryClient.invalidateQueries({ queryKey: ["joinRequests"] }); // Should be ["myAgencyJoinRequests"]
}
```

### Non-Async Invalidation for Critical Data
```tsx
// ❌ Incorrect: Not waiting for critical invalidation
onSuccess: () => {
  queryClient.invalidateQueries({ queryKey: ["myAgencyMemberships"] }); // Should await
}
```

## Migration Guide

When adding cache invalidation to existing mutations:

1. Identify the mutation's data impact
2. List all affected query keys from the registry
3. Add `onSuccess` callback with invalidation
4. Use `Promise.all` for multiple invalidations
5. Test affected surfaces refresh correctly

## Testing

Verify cache invalidation by:

1. Trigger the mutation
2. Check that affected queries refetch
3. Check that UI updates with fresh data
4. Check that unrelated queries are not refetched
5. Check that invalidation completes before next user action
