# Standard UI-002 — Z-Index & Stacking Context Standard

Before modifying any `z-index`, engineers must determine:

1. Is the element clipped by `overflow`?
2. Is it constrained by a stacking context?
3. Does it render inline or via portal?
4. Does a positioning strategy already exist?

Changing `z-index` is the **last** step, not the first.

## Z-Index Scale

The project maintains a documented z-index scale rather than arbitrary values like `z-[9999]`.

```text
0      Normal content
10     Sticky headers
20     Dropdowns (inline)
30     Floating panels
40     Side drawers
50     Modals
60     Toasts
70     Global overlays
```

## Usage Guidelines

### Normal Content (z-index: 0)
- Default document flow
- No z-index declaration needed
- Applies to most UI elements

### Sticky Headers (z-index: 10)
- Navbar, sticky table headers
- Must remain above content but below overlays
- Example: `<Navbar className="z-10" />`

### Dropdowns (z-index: 20)
- Inline dropdowns and menus
- Above headers, below panels
- Example: `<ul className="absolute z-20" />`

### Floating Panels (z-index: 30)
- Popovers, tooltips, date pickers
- Above dropdowns, below drawers
- Example: `<Popover className="z-30" />`

### Side Drawers (z-index: 40)
- Navigation drawers, settings panels
- Above panels, below modals
- Example: `<Sheet className="z-40" />`

### Modals (z-index: 50)
- Dialogs, forms, confirmations
- Above drawers, below toasts
- Example: `<Dialog className="z-50" />`

### Toasts (z-index: 60)
- Notifications, alerts
- Above modals, below global overlays
- Example: `<Toaster className="z-60" />`

### Global Overlays (z-index: 70)
- Loading screens, global backdrops
- Highest priority layer
- Example: `<div className="fixed inset-0 z-70" />`

## Diagnostic Steps

Before adding `z-index`, follow this sequence:

1. **Check for overflow clipping**
   ```js
   getComputedStyle(element).overflow
   getComputedStyle(element).overflowY
   ```
   If `hidden`, use portal rendering, not z-index.

2. **Check for stacking context**
   ```js
   // Elements that create stacking contexts:
   // - position: relative/absolute/fixed + z-index
   // - opacity < 1
   // - transform, filter, perspective
   // - will-change: transform/opacity
   ```
   Identify the nearest stacking context ancestor.

3. **Check rendering strategy**
   - Is the overlay inline or portal-rendered?
   - Portals escape stacking contexts; inline overlays are constrained.

4. **Check existing positioning**
   - Does a positioning strategy already exist?
   - Are you competing with another overlay at the same level?

## Anti-Patterns

### Arbitrary High Values
```tsx
// ❌ Incorrect: Arbitrary escalation
<div className="z-[9999]">Dropdown</div>
```

### Z-Index for Clipping
```tsx
// ❌ Incorrect: z-index doesn't solve overflow: hidden
<div className="relative overflow-hidden">
  <div className="absolute z-[9999]">Clipped</div>
</div>
```

### Z-Index Without Context
```tsx
// ❌ Incorrect: Adding z-index without checking stacking context
<div className="z-50">Dropdown</div>
```

## Correct Patterns

### Check First, Then Apply
```tsx
// ✅ Correct: Diagnostic before implementation
const parent = element.parentElement;
const overflow = getComputedStyle(parent).overflow;
if (overflow === 'hidden') {
  // Use portal, not z-index
  return createPortal(dropdown, document.body);
}
```

### Use Documented Scale
```tsx
// ✅ Correct: Documented z-index value
<Popover className="z-30" />
```

### Escalate Deliberately
```tsx
// ✅ Correct: Document escalation with reason
// Elevated above default drawer to prioritize user feedback
<Toaster className="z-65" />
```

## RealtorNet Specifics

- Navbar uses `z-10`
- Hero search dropdown uses portal rendering (no z-index escalation needed)
- Property card filters use `z-20` (inline dropdowns)
- Agency dashboard filters use `z-20` (inline dropdowns)
- Modals use `z-50`
- Toasts use `z-60`
- Global loading overlay uses `z-70`
